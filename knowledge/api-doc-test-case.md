# 接口文档与测试用例设计

## 竞赛关联

比赛第三部分第3题要求选手编写系统设计文档和测试用例，涉及API接口设计文档和功能测试用例。这是软件工程中不可或缺的文档编写能力考核。

## 核心技能

- **接口设计文档**：接口描述、请求参数、响应参数、示例
- **测试用例设计**：用例编号、预置条件、测试步骤、预期结果
- **文档规范**：Markdown 格式、表格排版
- **用例覆盖**：正常流程、异常流程、边界条件

### 2.1 用户登录

**接口描述**：用户通过用户名和密码登录系统，获取 Token。

**请求方式**：POST

**请求路径**：/user/login

**请求参数**：

| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| username | String | 是 | 用户名 |
| password | String | 是 | 密码 |

**请求示例**：
```
POST /user/login
Content-Type: application/x-www-form-urlencoded

username=admin&password=admin123
```

**响应参数**：

| 参数名 | 类型 | 说明 |
|--------|------|------|
| code | Integer | 状态码，0表示成功 |
| message | String | 提示信息 |
| data.token | String | 登录 Token |
| data.userId | Long | 用户 ID |
| data.username | String | 用户名 |
| data.realName | String | 真实姓名 |
| data.role | String | 角色 |

**成功响应示例**：
```json
{
  "code": 0,
  "message": "操作成功",
  "data": {
    "token": "a1b2c3d4e5f6...",
    "userId": 1,
    "username": "admin",
    "realName": "管理员",
    "role": "admin"
  }
}
```

**失败响应示例**：
```json
{
  "code": 1002,
  "message": "用户名或密码错误",
  "data": null
}
```

---

### 2.2 查询用户列表

**接口描述**：分页查询用户列表，支持关键词搜索和状态筛选。

**请求方式**：GET

**请求路径**：/user/list

**请求参数**：

| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| keyword | String | 否 | 搜索关键词 |
| status | Integer | 否 | 用户状态：0-禁用 1-启用 |
| role | String | 否 | 角色：admin / user |
| pageNum | Integer | 否 | 页码，默认1 |
| pageSize | Integer | 否 | 每页大小，默认10 |

**请求示例**：
```
GET /user/list?keyword=admin&status=1&pageNum=1&pageSize=10
Authorization: a1b2c3d4e5f6...
```

**响应参数**：

| 参数名 | 类型 | 说明 |
|--------|------|------|
| code | Integer | 状态码 |
| message | String | 提示信息 |
| data | Array | 用户列表 |
| data[].id | Long | 用户 ID |
| data[].username | String | 用户名 |
| data[].realName | String | 真实姓名 |
| data[].phone | String | 手机号 |
| data[].email | String | 邮箱 |
| data[].status | Integer | 状态 |
| data[].role | String | 角色 |
| data[].createTime | String | 创建时间 |
| total | Long | 总记录数 |
```

### 3. 测试用例编写规范

#### 3.1 用例编号规则

```
TC-{模块}-{序号}
例：TC-LOGIN-001、TC-LIST-001、TC-DEL-001
```

#### 3.2 测试覆盖要求

- **正常流程**：至少覆盖 1 个完整的成功场景
- **异常流程**：至少覆盖 3 种异常情况（参数错误、权限不足、业务规则冲突）
- **边界条件**：至少覆盖 2 种边界情况（空值、极限值、特殊字符）

#### 3.3 用例编写原则

1. **独立性**：每个用例可以独立执行，不依赖其他用例
2. **可重复**：相同输入总是产生相同输出
3. **单一职责**：每个用例只测试一个功能点
4. **可追溯**：用例编号与需求对应

## 重点内容

- 接口文档包含：接口描述、请求方式、请求参数、响应参数、示例
- 测试用例包含：编号、标题、前置条件、输入、预期结果
- 参数表格使用 Markdown 表格，清晰展示字段信息
- 成功和失败响应都要有示例
- 测试用例覆盖正常、异常、边界三种场景

## 注意事项

- 接口文档需要与代码同步更新
- 测试用例的预期结果要明确可验证
- 参数类型要准确，不能模糊（如"字符串"应写"String"）
- 测试用例编号要有规律，便于管理和查找

## 常见误区

- 误区：接口文档缺少请求示例，开发者无法快速调用
- 误区：测试用例只有正常流程，没有异常和边界
- 误区：预期结果描述模糊，如"返回成功"而非"返回 code=0"
- 误区：测试用例之间有依赖关系，无法独立执行
