# 前端联调最小知识

## 定位

本页只讲前端选手必须理解的后端、合约和链端知识。目标是让前端能正确调用接口、展示链上结果、完成截图交付，不要求以前端选手身份深入实现后端或合约。

示例为通用模型，不包含非公开训练材料细节。

## 前端需要看懂的后端约定

| 后端概念 | 前端要会什么 |
| --- | --- |
| 统一响应 | 知道 `code`、`data`、`message` 在哪里 |
| 登录接口 | 成功后取 token，并保存到约定位置 |
| 当前用户接口 | 刷新页面后能恢复用户信息和角色 |
| 分页接口 | 会取 `records/list` 和 `total` |
| 详情接口 | 会用行 id 查询完整信息 |
| 下载接口 | 会设置 Blob 下载和文件名 |
| 错误码 | token 失效时跳转登录，普通错误显示 message |

## 前端需要看懂的链上字段

| 字段 | 页面展示建议 |
| --- | --- |
| `txHash` | 单独展示，可复制 |
| `blockNumber` | 展示为区块高度 |
| `contractAddress` | 使用等宽字体或截断显示 |
| `deployStatus` | 用状态标签展示 |
| `chainTime` | 格式化为可读时间 |
| `errorMessage` | 失败时展示给用户，不只打印控制台 |

## 前端联调闭环

```text
登录 -> 保存 token -> 调列表接口 -> 进入详情 -> 展示链上字段 -> 下载或截图交付
```

## API 映射阅读法

如果工程中存在统一 API 表，优先按这个顺序查：

1. API 编号对应的 `uri`。
2. 请求方法 `get/post`。
3. 是否需要路径参数。
4. 是否需要 loading。
5. 响应数据结构。

不要绕开统一请求封装。绕开后可能丢 token、丢 loading、丢错误处理。

## 常见联调错误

- 前端传 `pageSize`，后端要 `size`。
- 前端取 `res.data.list`，实际返回 `records`。
- token 保存到 `localStorage`，拦截器从 `sessionStorage` 读取。
- 下载接口按 JSON 处理，导致文件乱码。
- 链上状态只在控制台打印，页面没有证据。
- 后端返回失败，前端仍打开成功弹窗。

## 前端自检

| 检查项 | 问自己 |
| --- | --- |
| 登录 | token 保存位置和拦截器一致吗 |
| 列表 | current、size、keyword 是否传对 |
| 详情 | id 是否来自当前行 |
| 下载 | 是否使用 Blob，是否释放 Object URL |
| 状态 | loading 是否一定关闭 |
| 错误 | catch 是否有用户提示 |
| 链上证据 | 页面是否能看到交易或部署结果 |
